1. 目标与边界
建立统一的任务空间（笛卡尔）运动控制能力，支持 move_p 与 move_l 两类运动语义。
保持现有前端事件协议尽量兼容，但新增明确语义的事件。
将“运动学计算”和“运动执行”彻底解耦，避免前端操作直接触发零散关节命令。
明确坐标系、单位、速度约束，消除“看起来点了但不动/不对”的体验。
当前阶段不做复杂避障与碰撞规划，只做单臂/双臂基础运动控制。
2. 核心概念统一（必须先定）
MotionType
JOINT_PTP：关节点到点
CARTESIAN_PTP：笛卡尔点到点（对应 move_p）
CARTESIAN_LINEAR：笛卡尔直线（对应 move_l）
CARTESIAN_JOG：笛卡尔点动/连续点动
Frame
BASE：基座坐标系（推荐默认）
TOOL：工具坐标系
USER：用户工件坐标系（后续扩展）
Pose
{x,y,z,rx,ry,rz}，单位统一：位置 m，姿态 rad
Constraints
max_velocity, max_acceleration, max_angular_velocity, max_angular_acceleration
关节限位、关节速度/加速度限位为硬约束
ExecutionMode
blocking | non_blocking（move_l按 SDK 特性可内部强制 blocking，但对外统一接口）
3. move_p / move_l 行为定义（最终口径）
move_p（CARTESIAN_PTP）

输入 TCP 6D 位姿。
由 IK 求解目标关节位形，执行关节空间轨迹。
末端路径不保证直线。
优点：快、收敛性好、对实时性压力较低。
move_l（CARTESIAN_LINEAR）

输入 TCP 6D 位姿。
末端按笛卡尔直线轨迹运动，位置梯形速度规划，姿态 Slerp。
每个采样点必须可解且满足约束，轨迹跟踪严格。
优点：工艺路径可控；缺点：更容易触发奇异点/限位/不可达。
4. 后端模块架构（建议新增）
建议在 robot_arm_web/backend/ 下拆分：

motion_types.py
定义 MotionCommand, MotionResult, ErrorCode, Frame, MotionType
frame_manager.py
管理 BASE/TOOL/USER 坐标变换，统一 frame 入口
cartesian_planner.py
plan_move_p(...), plan_move_l(...), plan_jog(...)
trajectory_executor.py
控制周期执行，支持暂停/恢复/取消/抢占
motion_service.py
对外统一入口，供 app.py socket 事件调用
motion_monitor.py
跟踪误差、状态广播、故障上报
现有 app.py 的事件函数应只做参数校验+转发，不做规划细节。

5. 对现有代码的映射建议
backend/app.py
保留现有事件名兼容（如 cartesian_move_to, cartesian_jog, smooth_cartesian_move）
内部改为调用 motion_service.execute(command)
backend/cartesian_controller.py
定位为“运动学工具层”（FK/IK），不直接承担执行
backend/group_controller.py
作为硬件适配层，提供整臂原子下发接口（已具备基础）
不再承载上层轨迹策略判断
6. API/事件协议（兼容+升级）
6.1 新增统一命令事件（建议）
motion:execute
motion:cancel
motion:pause
motion:resume
motion:query
请求示例：

{
  "command_id": "uuid",
  "arm_id": "left",
  "motion_type": "CARTESIAN_LINEAR",
  "target_pose": {"x":0.12,"y":0.03,"z":0.88,"rx":0.0,"ry":-1.57,"rz":0.0},
  "frame": "BASE",
  "constraints": {
    "max_velocity": 0.05,
    "max_acceleration": 0.1,
    "max_angular_velocity": 0.3,
    "max_angular_acceleration": 0.1
  },
  "blocking": true
}
返回/广播示例：

{
  "command_id": "uuid",
  "status": "running",
  "progress": 0.42,
  "pose_error_mm": 0.8,
  "joint_error_rad": 0.01
}
6.2 旧事件兼容映射
cartesian_move_to -> 映射为 motion_type=CARTESIAN_PTP
smooth_cartesian_move -> 可临时映射为 CARTESIAN_LINEAR（仅位置）
cartesian_jog -> 映射为 CARTESIAN_JOG
7. 前端界面改造设计（你特别关心的部分）
涉及文件：frontend/index.html, frontend/js/main.js, frontend/css/style.css

7.1 笛卡尔页重构为 4 个子模块
目标位姿输入区
x y z rx ry rz
frame 选择：BASE/TOOL
单位显示与转换（UI mm/deg，后端 m/rad）
运动模式区

执行状态区

7.2 关键交互要求
用户每次下发前必须看到：当前 arm_id、frame、motion_type
不可达时提示具体原因：IK_UNREACHABLE、SINGULARITY_NEAR、JOINT_LIMIT
执行中参数输入框置灰，避免并发误操作
日志区分“命令已接收”与“命令已完成”

11. 验收标准（必须量化）
move_p：95% 以上指令在可达区成功执行
move_l：直线段最大偏差 < 2mm（按你设备能力可再调）
前端控制响应：事件发出到状态变化 < 100ms
错误可解释率：100% 有错误码与中文解释
无“点了无反馈”黑洞状态
